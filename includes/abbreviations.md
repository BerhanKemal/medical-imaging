<!-- DART LANGUAGE                                               -->
<!-- ═══════════════════════════════════════════════════════════ -->

*[Dart]: The programming language behind Flutter. Dart is developed by Google, supports ahead-of-time (AOT) compilation for fast mobile apps, and just-in-time (JIT) compilation for hot reload during development. It features sound null safety, async/await, and a C-style syntax familiar to Java/C# developers.
*[null safety]: A Dart language feature (introduced in Dart 2.12) that eliminates null reference errors at compile time. Variables are non-nullable by default. Use ? to allow null (String?), ! to assert non-null, ?? for default values, and ?. for safe access. This prevents the "billion dollar mistake" of null pointer exceptions.
*[async/await]: Dart's syntax for writing asynchronous code that reads like synchronous code. Mark a function 'async' to use 'await' inside it. 'await' pauses execution until a Future completes, but does NOT block the UI thread — Flutter keeps rendering while waiting. Essential for database queries, API calls, and file I/O.
*[Future]: A Dart object representing a value that will be available later — the result of an asynchronous operation. Future<String> means "a String that isn't ready yet." Use 'await' to get the value, or .then() for callback style. Common in API calls (Future<Response>), database queries (Future<List<Map>>), and file operations.
*[Stream]: A Dart object that delivers a sequence of asynchronous values over time — like a Future that can emit multiple results. Used for real-time data: WebSocket messages, sensor readings, Firebase snapshots, or BLE device data. Listen with 'await for' or StreamBuilder widget.
*[typedef]: A Dart keyword for creating type aliases — giving a name to a function signature or complex type. Example: typedef JsonMap = Map<String, dynamic>. Makes code more readable when passing callbacks or working with deeply nested generic types.
*[mixin]: A Dart mechanism for reusing code across multiple class hierarchies without inheritance. Declared with 'mixin' keyword, applied with 'with'. Common in Flutter: TickerProviderStateMixin for animations, WidgetsBindingObserver for lifecycle events.
*[extension]: A Dart feature (Dart 2.7+) that lets you add methods to existing classes without modifying them. Example: extension on DateTime { String toShortString() => ... }. Useful for adding domain-specific helpers to built-in types.
*[sealed class]: A Dart 3 feature for creating closed class hierarchies — the compiler knows all possible subtypes, enabling exhaustive switch expressions. Perfect for modeling finite states like AuthState (authenticated, unauthenticated, loading, error).
*[record]: A Dart 3 feature for lightweight, immutable data structures — like tuples with optional named fields. Example: (String name, int age) or ({String name, int age}). Useful for returning multiple values from a function without creating a class.
*[pattern matching]: A Dart 3 feature for destructuring and testing values in switch expressions, if-case statements, and variable declarations. Enables exhaustive type checking and cleaner control flow compared to if-else chains.

<!-- FLUTTER FRAMEWORK                                           -->
<!-- ═══════════════════════════════════════════════════════════ -->

*[Flutter]: Google's open-source UI toolkit for building natively compiled applications for mobile, web, and desktop from a single Dart codebase. Flutter uses its own rendering engine (Impeller on iOS, Skia fallback) — it does NOT use native UI components, giving pixel-perfect control across platforms.
*[Widget]: The fundamental building block of Flutter UI. Everything visible on screen is a Widget — an immutable description of part of the interface. Widgets form a tree: parent widgets contain child widgets. When state changes, Flutter rebuilds the widget tree and efficiently updates only what changed on screen. Widgets are cheap to create.
*[StatelessWidget]: A Flutter widget with no mutable state — its build() method depends only on the constructor arguments. Rebuilt from scratch when its parent passes new data. Use for static content (labels, icons) or pure-function UI that transforms input data into widgets. Simpler and more performant than StatefulWidget.
*[StatefulWidget]: A Flutter widget that maintains mutable state across rebuilds via a companion State object. The State object persists when the widget tree rebuilds, allowing animations, form input, and local counters. Call setState() to trigger a rebuild. Consider replacing with Riverpod providers for shared state.
*[ConsumerWidget]: A Riverpod-specific replacement for StatelessWidget that provides a WidgetRef parameter in its build() method. This ref lets you call ref.watch() to subscribe to providers and ref.read() for one-time access. Use ConsumerWidget for any screen or component that needs to access Riverpod state.
*[ConsumerStatefulWidget]: A Riverpod-specific replacement for StatefulWidget that provides ref access in both build() and lifecycle methods (initState, dispose). Use when you need both mutable local state AND provider access — for example, TextEditingControllers combined with API calls.
*[BuildContext]: A handle to a widget's location in the widget tree. Used to look up inherited data (Theme.of(context), MediaQuery.of(context)), show dialogs (showDialog(context: context)), and navigate (Navigator.of(context).push(...)). Each widget has its own BuildContext. Never store it across async gaps.
*[Scaffold]: Material Design's page-level layout structure. Provides slots for AppBar (top bar), body (main content), FloatingActionButton, Drawer (side menu), BottomNavigationBar, and SnackBar. Most screens in a Material app have a Scaffold as their root widget. Handles safe areas and keyboard insets automatically.
*[MaterialApp]: The root widget of a Material Design Flutter app. Configures theming (colors, typography), routing (named routes or GoRouter), localization (multiple languages), and debug settings. Wraps your widget tree with Navigator, Theme, and MediaQuery. There should be exactly one MaterialApp in your app.
*[Navigator]: Flutter's stack-based routing system. Navigator.push() adds a new screen on top of the stack (like opening a page). Navigator.pop() removes the current screen (like pressing back). Navigator 2.0 / GoRouter adds declarative, URL-based routing for more complex navigation patterns.
*[setState]: The method that tells Flutter "my state changed, please rebuild this widget." Calling setState() inside a State object schedules a call to build(). Only use inside StatefulWidget. For shared state across widgets, prefer Riverpod providers instead — they provide the same reactivity without the tight coupling.
*[BuildContext]: A handle to a widget's position in the tree. Passed to build() and used to access ancestors: Theme.of(context) for colors, Navigator.of(context) for routing, MediaQuery.of(context) for screen size. Each widget gets its own context. Never use a context after its widget is disposed.
*[pubspec.yaml]: The configuration file at the root of every Dart/Flutter project. Declares the project name, version, Dart SDK constraints, dependencies (from pub.dev), dev dependencies (testing tools), and asset paths (images, fonts). Run 'flutter pub get' after editing to download new packages.
*[hot reload]: Flutter's signature developer experience feature. Press r (or save in IDE) to inject updated Dart source code into the running app in under a second, WITHOUT losing current state. Form inputs, scroll positions, and navigation state are preserved. Works because Dart's VM supports incremental compilation.
*[hot restart]: A full restart of the Flutter app that resets all state to initial values. Use Shift+R when hot reload doesn't pick up changes — typically after modifying main(), changing static fields, or updating enum definitions. Faster than a full rebuild but loses all in-memory state.
*[DevTools]: Flutter's browser-based suite of debugging and performance tools. Includes: Widget Inspector (explore the widget tree), Performance View (identify jank), Network Profiler (track HTTP requests), Memory Profiler (find leaks), and Logging View. Launch from VS Code, Android Studio, or command line.
*[InheritedWidget]: Flutter's built-in mechanism for passing data down the widget tree without explicitly threading it through constructors. Theme, MediaQuery, and Navigator all use InheritedWidget internally. Riverpod and Provider are higher-level abstractions built on this same concept.
*[Key]: An optional identifier that Flutter uses to match widgets across rebuilds. Without keys, Flutter matches by type and position. Use ValueKey for list items (to preserve state during reorder), GlobalKey for accessing widget state from outside, and UniqueKey to force recreation.
*[WidgetRef]: The Riverpod object passed to ConsumerWidget.build() that provides access to providers. Call ref.watch(provider) to subscribe and rebuild on changes. Call ref.read(provider) for one-time reads in callbacks. Call ref.listen(provider, callback) for side effects without rebuilding.

<!-- RIVERPOD STATE MANAGEMENT                                   -->
<!-- ═══════════════════════════════════════════════════════════ -->

*[Riverpod]: A reactive, compile-safe state management framework for Flutter by Remi Rousselet. Unlike Provider (its predecessor), Riverpod providers are declared as global variables and are independent of the widget tree. This eliminates ProviderNotFoundException errors and enables testing without widget pumping. Version 2.0+ uses code generation for type safety.
*[Provider]: In Riverpod, a declaration that exposes a read-only computed value to the widget tree. Created with Provider<T>((ref) => value). The value is lazily computed and cached. Use for derived/computed data (e.g., filtered lists, formatted strings) or service instances. Not to be confused with the older 'provider' package.
*[StateNotifier]: A Riverpod class (from the state_notifier package) that holds a single immutable state value and provides methods to update it. The state setter automatically notifies all listeners. Think of it as a ViewModel in MVVM architecture. Always create a new state object instead of mutating — use state = [...state, newItem] not state.add(newItem).
*[StateNotifierProvider]: A Riverpod provider that creates a StateNotifier and exposes both the notifier (for calling methods) and its current state (for reading values). Access the state with ref.watch(provider), access the notifier with ref.read(provider.notifier). The most common provider type for mutable business logic.
*[ProviderScope]: The root widget that stores all Riverpod provider state. Must wrap your MaterialApp — without it, ref.watch() throws. Think of it as a "state container" that lives above your entire widget tree. You can create nested ProviderScopes for testing or overriding specific providers.
*[ref.watch]: Subscribes to a Riverpod provider and automatically rebuilds the widget whenever the provider's value changes. Use ONLY inside build() methods. Creates a reactive dependency: Widget → Provider. When the provider updates, Flutter efficiently rebuilds just the widgets that watch it. The backbone of reactive UI.
*[ref.read]: Reads a Riverpod provider's current value exactly once, without subscribing to future changes. Use in event handlers, callbacks, onPressed, and initState — anywhere that runs once in response to user action. Using ref.read() in build() is a bug — the widget won't update when the provider changes.
*[ref.listen]: Registers a callback that fires whenever a Riverpod provider changes, without triggering a widget rebuild. Use for side effects: showing SnackBars on error, navigating on auth state change, or logging. Takes two parameters: the provider and a callback(previous, next). Call inside build() or ConsumerStatefulWidget.initState().
*[ref.invalidate]: Forces a Riverpod provider to recompute its value on next access. Useful for "pull to refresh" — invalidate the data provider and it will re-fetch from the API. More targeted than setState: only the invalidated provider and its dependents recompute.
*[FutureProvider]: A Riverpod provider for asynchronous operations that produce a single value — like API calls or database queries. Automatically wraps the result in AsyncValue (data/loading/error), making it easy to show loading spinners and error messages in the UI without manual state management.
*[AsyncValue]: Riverpod's wrapper for asynchronous data with three states: AsyncData (value ready), AsyncLoading (waiting), and AsyncError (failed). Use .when(data: ..., loading: ..., error: ...) for exhaustive handling in build methods. Eliminates the need for manual isLoading/hasError boolean flags.

<!-- DATA & PERSISTENCE                                          -->
<!-- ═══════════════════════════════════════════════════════════ -->

*[SQLite]: A lightweight, serverless, file-based relational database engine. In Flutter, accessed via the sqflite package. The entire database is a single file on the device. Supports SQL queries, transactions, and indices. Perfect for offline-first mobile apps — data persists across app restarts. Limited to ~1GB practically on mobile.
*[sqflite]: The most popular Flutter plugin for SQLite database access. Provides async methods for CRUD operations, raw SQL queries, batch operations, and schema migrations via version numbers. Works on iOS, Android, and macOS. Use with the path package to locate the database file in the app's documents directory.
*[CRUD]: Create, Read, Update, Delete — the four fundamental operations for persistent data. Maps to SQL (INSERT, SELECT, UPDATE, DELETE) and HTTP methods (POST, GET, PUT, DELETE). Every data-driven feature in your app ultimately breaks down into combinations of these four operations.
*[Repository pattern]: A design pattern that provides an abstraction layer between your business logic and data sources. The repository exposes clean methods like getMoods() and addMood() while hiding whether data comes from SQLite, a REST API, or both. This makes it easy to swap storage backends, add caching, or implement offline-first sync.
*[Singleton pattern]: A design pattern that ensures only one instance of a class exists throughout the app's lifetime. In Flutter, used for database connections (DatabaseHelper.instance) and service classes. Implemented with a private constructor and a static final field. Prevents multiple database connections competing for the same file.
*[ORM]: Object-Relational Mapping — a technique that lets you work with database rows as Dart objects instead of raw Maps. Your toMap()/fromMap() methods are manual ORM. Packages like drift (formerly moor) provide full ORM with type-safe queries, auto-generated code, and migration support. Trade-off: convenience vs control.
*[JSON]: JavaScript Object Notation — the standard data format for REST APIs. A JSON object { "key": "value" } maps directly to Dart's Map<String, dynamic>. Arrays map to List. Use jsonEncode() to convert Dart to JSON string, jsonDecode() to parse JSON string to Dart. Field names are typically snake_case in APIs and camelCase in Dart.
*[Serialization]: The process of converting a Dart object into a transferable format — typically Map<String, dynamic> for JSON APIs or SQLite storage. Implemented as toJson() or toMap() instance methods. Must handle type conversions: DateTime to ISO 8601 string, enums to strings, nested objects to nested maps. The reverse is deserialization.
*[Deserialization]: The reverse of serialization — converting raw data (JSON string, database row, Map) back into a typed Dart object. Typically implemented as a factory constructor: factory MoodEntry.fromJson(Map<String, dynamic> json). Must handle null fields, type parsing (String to DateTime), and missing keys gracefully.
*[ISO 8601]: The international standard format for representing dates and times as strings: "2025-03-15T14:30:00.000Z". Used by most REST APIs and required for SQLite (which has no native DateTime type). In Dart: DateTime.now().toIso8601String() to encode, DateTime.parse(str) to decode. The trailing Z means UTC.
*[migration]: In databases, a migration is a versioned change to the schema (adding tables, columns, or indices). In sqflite, controlled by the version parameter in openDatabase() and the onUpgrade callback. Each version number corresponds to a schema change. Never modify a released migration — add a new version instead.
*[transaction]: A database operation that groups multiple queries into an atomic unit — either ALL succeed or ALL are rolled back. Use db.transaction((txn) => ...) in sqflite when inserting related data across multiple tables. Prevents partial writes that leave the database in an inconsistent state. Critical for financial and health data integrity.
*[SharedPreferences]: A Flutter plugin for storing simple key-value pairs (strings, ints, bools) in platform-native storage. Fast but NOT encrypted and NOT suitable for sensitive data like tokens or passwords. Use for user preferences (theme, language, onboarding status). For sensitive data, use FlutterSecureStorage instead.

<!-- NETWORKING & API                                            -->
<!-- ═══════════════════════════════════════════════════════════ -->

*[API]: Application Programming Interface — a contract defining how two software systems communicate. In this course, we use REST APIs: the Flutter app sends HTTP requests to a server, which returns JSON responses. The API defines endpoints (URLs), accepted methods (GET/POST), request formats, and response structures. Think of it as a waiter between the kitchen (server) and you (app).
*[REST]: Representational State Transfer — the most common architectural style for web APIs. REST principles: resources have URLs (/moods, /moods/42), HTTP methods define actions (GET=read, POST=create, PUT=update, DELETE=remove), responses are stateless (each request carries all needed info), and data is typically JSON. Designed by Roy Fielding in 2000.
*[HTTP]: HyperText Transfer Protocol — the foundation of communication on the web. An HTTP request has: a method (GET/POST/PUT/DELETE), a URL, headers (metadata like Content-Type and Authorization), and optionally a body (for POST/PUT). The server responds with a status code (200=OK, 404=Not Found, 500=Server Error), headers, and a body.
*[HTTPS]: HTTP Secure — HTTP encrypted with TLS (Transport Layer Security). All data between the app and server is encrypted in transit, preventing eavesdropping, tampering, and man-in-the-middle attacks. Required by Apple (App Transport Security) and Google Play for production apps. Always use https:// URLs for APIs, never http://.
*[GET]: The HTTP method for retrieving data from a server. GET requests should be idempotent (calling twice produces the same result) and should NEVER modify server data. Parameters go in the URL query string (?page=1&limit=20), not in the body. Example: GET /api/moods returns all mood entries.
*[POST]: The HTTP method for creating a new resource on the server. The data to create goes in the request body, usually as JSON. The server typically returns the created resource (with server-assigned ID and timestamps) and status 201 Created. Example: POST /api/moods with body {"score": 8, "note": "Great day"}.
*[PUT]: The HTTP method for replacing an existing resource on the server. The entire updated resource goes in the body. The URL includes the resource ID: PUT /api/moods/42. Returns the updated resource. Idempotent: calling PUT twice with the same data produces the same result. Use PATCH for partial updates.
*[DELETE]: The HTTP method for removing a resource from the server. The URL includes the resource ID: DELETE /api/moods/42. Returns status 204 No Content on success. Should be idempotent: deleting an already-deleted resource returns success (not error). In healthcare apps, consider soft-delete (marking as deleted) instead of hard-delete for audit trails.
*[JWT]: JSON Web Token — a compact, self-contained token format for authentication. Structure: header.payload.signature (base64-encoded, dot-separated). The payload contains claims (user ID, expiration). The signature prevents tampering. The server issues a JWT on login; the client sends it in every subsequent request's Authorization header. Stateless: the server doesn't need to store sessions.
*[OAuth]: Open Authorization — an industry-standard protocol for delegated access. Instead of sharing passwords, users authorize apps to access their data via tokens. OAuth2 defines several "flows" (authorization code, password grant, client credentials). The password grant (used in this course) exchanges username+password for a token. In production, prefer authorization code flow with PKCE.
*[Bearer token]: An access token sent in the HTTP Authorization header: "Authorization: Bearer eyJhbG...". "Bearer" means the server trusts whoever presents this token — no additional proof of identity needed. This makes token storage security critical: if an attacker steals the token, they can impersonate the user. Always store in FlutterSecureStorage, never in SharedPreferences.
*[CORS]: Cross-Origin Resource Sharing — a browser security mechanism that restricts which domains can call an API. The server sets Access-Control-Allow-Origin headers to whitelist origins. Primarily a web concern: mobile apps are NOT affected by CORS because they don't run in a browser sandbox. You may encounter CORS during Flutter Web development.
*[status code]: A three-digit number in every HTTP response indicating the result. 2xx = success (200 OK, 201 Created, 204 No Content). 3xx = redirect. 4xx = client error (400 Bad Request, 401 Unauthorized, 404 Not Found, 422 Validation Error). 5xx = server error (500 Internal Server Error). Always check status codes before parsing response bodies.
*[endpoint]: A specific URL path on a server that handles a particular operation. Example: /api/moods is the endpoint for mood CRUD operations. GET /api/moods returns all moods, POST /api/moods creates one. The base URL (https://server.com) + endpoint (/api/moods) = full URL. Design endpoints around resources (nouns), not actions (verbs).
*[Content-Type]: An HTTP header that tells the receiver what format the body is in. For JSON APIs: "application/json". For form data (OAuth2 login): "application/x-www-form-urlencoded". For file uploads: "multipart/form-data". If you send JSON with the wrong Content-Type, the server may reject or misparse the request.
*[base URL]: The root address of an API server (e.g., https://api.example.com/v1). All endpoints are relative to this base. Store it as a constant or environment variable — never hardcode full URLs for each request. This makes it easy to switch between development (localhost), staging, and production servers.

<!-- AUTHENTICATION & SECURITY                                   -->
<!-- ═══════════════════════════════════════════════════════════ -->

*[FlutterSecureStorage]: A Flutter plugin for storing sensitive data (tokens, passwords, encryption keys) using platform-native secure storage. On iOS: Keychain Services (hardware-backed encryption). On Android: EncryptedSharedPreferences (AES-256-SIV with Android Keystore master key). Data persists across app restarts but is deleted on app uninstall. Always use this instead of SharedPreferences for secrets.
*[Keychain]: Apple's secure storage service for sensitive data on iOS and macOS. Data is encrypted at rest, protected by the device passcode, and optionally requires biometric authentication (Face ID/Touch ID) to access. Each app has its own sandboxed keychain entries. FlutterSecureStorage uses Keychain internally on Apple platforms.
*[Keystore]: Android's hardware-backed cryptographic key storage system. Cryptographic keys are stored in a secure element (hardware chip) and never leave it — even the app can't extract the raw key. EncryptedSharedPreferences uses Keystore to protect the AES master key. The Android equivalent of Apple's Keychain.
*[PBKDF2]: Password-Based Key Derivation Function 2 — an algorithm that converts a user's password into a cryptographic key by applying a hash function (e.g., SHA-256) thousands of times with a random salt. The iteration count makes brute-force attacks computationally expensive. Used for password storage (never store plain text passwords) and deriving encryption keys.
*[Biometric authentication]: Verifying a user's identity using unique physical characteristics — fingerprint (Touch ID / Android Fingerprint), face geometry (Face ID), or iris scan. In Flutter, the local_auth package provides a cross-platform API. Biometrics are a "something you are" factor, complementing passwords ("something you know") for multi-factor authentication.
*[AES-256]: Advanced Encryption Standard with 256-bit key length — a symmetric encryption algorithm widely considered unbreakable with current technology. Used by FlutterSecureStorage (via EncryptedSharedPreferences on Android) and by governments worldwide for classified information. "Symmetric" means the same key encrypts and decrypts.
*[TLS]: Transport Layer Security — the cryptographic protocol that powers HTTPS. Establishes an encrypted channel between client and server using asymmetric key exchange (RSA/ECDH) then symmetric encryption (AES). Prevents eavesdropping, tampering, and impersonation. Flutter's http package uses TLS automatically for https:// URLs.
*[auth state]: The current authentication status of the user in the app. Typically modeled as a sealed class or enum with states: initial (app just launched), loading (checking token or logging in), authenticated (valid token exists), unauthenticated (no token), and error (login failed). The UI renders different screens based on this state.
*[access token]: A credential (usually a JWT) that the client sends with every API request to prove the user's identity. Has a limited lifetime (minutes to hours). When it expires, the client must refresh it using a refresh token or re-authenticate. Stored securely in FlutterSecureStorage, sent in the Authorization header.
*[refresh token]: A long-lived credential used to obtain a new access token without re-entering credentials. Stored securely alongside the access token. When the access token expires, the client sends the refresh token to a special endpoint to get a fresh pair. If the refresh token is compromised, the attacker gets persistent access — protect it carefully.

<!-- HEALTHCARE & COMPLIANCE                                     -->

*[mHealth]: Mobile Health — the practice of using mobile devices, apps, wearables, and sensors for healthcare delivery, patient monitoring, clinical research, and public health. Examples: fitness trackers, remote patient monitoring, medication reminders, teleconsultation apps. The global mHealth market is projected to exceed $300 billion by 2030.
*[HIPAA]: Health Insurance Portability and Accountability Act (1996, USA) — a federal law that requires organizations handling Protected Health Information (PHI) to implement physical, technical, and administrative safeguards. Violations can result in fines from $100 to $1.5 million per incident. Key requirements: encryption at rest and in transit, access controls, audit logging, and Business Associate Agreements.
*[GDPR]: General Data Protection Regulation (2018, EU) — the world's strictest data privacy law. Applies to ANY app processing EU residents' data, regardless of where the company is based. Key requirements: explicit consent, right to erasure ("right to be forgotten"), data portability, breach notification within 72 hours, and Privacy by Design. Fines up to 4% of global revenue.
*[PHI]: Protected Health Information — any individually identifiable health data under HIPAA. Includes: patient names, addresses, dates (birth, admission, discharge), phone numbers, email, Social Security numbers, medical record numbers, biometric identifiers, and ANY health condition or treatment data linked to an individual. De-identified data (18 identifiers removed) is NOT PHI.
*[FHIR]: Fast Healthcare Interoperability Resources (pronounced "fire") — a modern standard by HL7 for exchanging healthcare data via RESTful APIs. Resources (Patient, Observation, MedicationRequest) are like database tables with standardized fields. Supports JSON and XML. Used by Apple Health Records, Google Health API, and most modern EHR systems. Designed to be developer-friendly.
*[HL7]: Health Level Seven International — the organization that develops standards for healthcare data exchange. HL7v2 (legacy, pipe-delimited messages) is still used in most hospitals. HL7v3/CDA (XML-based) was complex and saw limited adoption. FHIR (2014+) is the modern successor combining the best of v2's simplicity and v3's rigor.
*[FDA]: Food and Drug Administration (USA) — the agency that regulates medical devices, including certain mobile health apps. If your app diagnoses, treats, or prevents disease (e.g., ECG analysis, insulin dosing), it's regulated as a Software as a Medical Device (SaMD). Wellness apps (step counters, meditation) are generally exempt. Classification: Class I (low risk) to Class III (life-sustaining).
*[MDR]: Medical Device Regulation (EU 2017/745) — replaced the older Medical Device Directives in May 2021. Stricter classification rules for software: any app that provides information used for clinical decision-making is at least Class IIa. Requires: clinical evaluation, post-market surveillance, UDI (Unique Device Identification), and a CE marking. Enforced by Notified Bodies.
*[SaMD]: Software as a Medical Device — software intended for medical purposes that is not part of a hardware medical device. Examples: an app that analyzes skin lesion photos for melanoma risk, software that calculates insulin doses, or an AI that reads X-rays. Classification depends on the seriousness of the condition and the significance of the software's output to clinical decisions.
*[DiGA]: Digitale Gesundheitsanwendungen (Digital Health Applications) — a German fast-track regulatory pathway where certified health apps can be prescribed by doctors and reimbursed by statutory health insurance (GKV). Requirements: CE marking as Class I/IIa medical device, data privacy compliance, interoperability (FHIR), and a 12-month evaluation period to prove positive healthcare effects.
*[IEC 62304]: International standard for the software development lifecycle of medical device software. Requires: software development planning, requirements analysis, architectural design, detailed design, unit testing, integration testing, and maintenance procedures. Software is classified into Class A (no injury possible), B (non-serious injury), or C (death or serious injury). Higher classes require more rigorous documentation.
*[CE marking]: Conformité Européenne — a mandatory marking for products sold in the European Economic Area. For medical devices, it means the product has been assessed by a Notified Body and meets EU safety, health, and performance requirements. Not a quality mark — it's a legal declaration of conformity. Required before placing a medical device on the EU market.
*[EHR]: Electronic Health Record — a digital version of a patient's medical chart. Contains medical history, diagnoses, medications, lab results, immunizations, and treatment plans. Major systems: Epic, Cerner (Oracle Health), MEDITECH. Modern EHRs expose FHIR APIs that mobile apps can integrate with to read/write patient data (with proper authorization).
*[clinical decision support]: Software that provides clinicians with knowledge and patient-specific information to enhance decision-making. Examples: drug interaction alerts, diagnostic suggestions, risk calculators. If your app does this, it's likely SaMD and subject to FDA/MDR regulation. Distinguish from "clinical decision support exempt" tools that merely display unprocessed data.
*[informed consent]: The process of obtaining a patient's or research participant's voluntary agreement to participate, after explaining the purpose, procedures, risks, benefits, and alternatives. In mHealth apps: must be obtained digitally with clear language, stored with timestamps, and allow withdrawal at any time. Required by HIPAA, GDPR, and research ethics boards (IRB/EC).

<!-- ACCESSIBILITY                                               -->

*[WCAG]: Web Content Accessibility Guidelines — developed by W3C, these international standards define three conformance levels: A (minimum), AA (standard, legally required in many jurisdictions), and AAA (enhanced). Key principles: Perceivable, Operable, Understandable, Robust (POUR). WCAG 2.1 AA is the baseline for mobile health apps.
*[VoiceOver]: Apple's built-in screen reader for iOS, iPadOS, and macOS. Reads aloud UI elements in order, lets users navigate by swiping, and announces semantic information (button, heading, image description). Flutter's Semantics widget controls what VoiceOver announces. Test your app with VoiceOver: Settings > Accessibility > VoiceOver on iOS.
*[TalkBack]: Google's built-in screen reader for Android. Equivalent to Apple's VoiceOver — provides spoken feedback, haptic responses, and gesture-based navigation for visually impaired users. Flutter widgets automatically generate TalkBack-compatible accessibility trees. Test with: Settings > Accessibility > TalkBack on Android.
*[Semantics]: In Flutter, a widget that annotates its child with accessibility metadata: labels (what it is), hints (what it does), values (current value), traits (button, heading, image), and actions (tap, long-press). Screen readers like VoiceOver and TalkBack read this metadata aloud. Every interactive widget should have meaningful semantic labels.
*[a11y]: Numeronym for "accessibility" — the letter 'a', followed by 11 letters, followed by 'y'. A shorthand used in development communities. In Flutter, accessibility features include: Semantics widgets, sufficient color contrast, minimum touch target sizes (48x48dp), dynamic text scaling support, and screen reader compatibility.
*[contrast ratio]: The relative luminance difference between foreground text and background color, expressed as a ratio (e.g., 4.5:1). WCAG AA requires 4.5:1 for normal text and 3:1 for large text. Low contrast makes text unreadable for users with low vision or in bright sunlight. Use tools like the WebAIM Contrast Checker to verify.
*[touch target]: The tappable area of an interactive UI element. WCAG and Material Design require a minimum of 48x48 density-independent pixels (dp) to accommodate users with motor impairments or large fingers. Flutter's InkWell and GestureDetector can extend touch targets beyond the visible widget bounds using padding.
